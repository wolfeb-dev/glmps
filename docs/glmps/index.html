GLMPS Documentation

A local, per-session observability and self-improvement control plane for AI-agent harnesses. It watches every session, surfaces it live, and closes a loop that turns observed mistakes into guard rules.

observe any harness · act only on Claude Code · zero runtime deps

01Overview

GLMPS (internally "GLMPS") tails the transcript and log files that AI coding agents write — Claude Code, Antigravity, Gemini CLI, Codex CLI, Cline, and others — classifies their raw output into a single shared event stream, and serves it in a real-time dashboard at http://127.0.0.1:8123.

Beyond passive observation it closes a feedback loop: when a session ends, a Stop hook re-parses the transcript, detects capability misuses (not reaching for a skill, running a backtest without validation, re-reading the same file in a loop), records those gaps, and can synthesize new CLAUDE.md guard rules from accumulated data. The observation plane is tool-agnostic; the control plane is scoped to the one harness that can be safely actuated.

The mental model

  • Observe any harness. Ten capture adapters (plus detect-only entries for five more) normalize every tool into one event shape.
  • Act on Claude Code only. Resume, terminal launch, queue-runner execution, and guard application are gated on the one adapter marked controllable. Foreign harnesses are structurally observe-and-recommend.
  • Logical, not physical, consolidation. GLMPS indexes across two planes — the user's repos and the agents' tool directories — wherever those files live. It never relocates them.

Design philosophy

Zero runtime depsserver/ and web/ use only Node built-ins. Auditable, no supply-chain surface.
No build stepweb/ is served from disk per request — edits are live on refresh.
Adapter-basedA new tool is one file in adapters/; the core loop is untouched.
Capability-firstThe project exists to catch its own misuse and prevent it over time.
Dev/prod separationProd surfaces are never hand-edited; crossings go through a gate.
TDDEvery server/lib change ships with node:test coverage.

02Deployment

Declared in server/lib/deps.js and verified by glmps init.

Install

# 1. Clone and enter the repo
git clone <repo-url> glmps
cd glmps

# 2a. Full setup: generate a per-client profile AND wire everything
node scripts/glmps-init.mjs --engagement <name> [--search <dir>] [--handle <handle>] [--force]

# 2b. OR just wire the base installer (no profile)
npm run setup            # = node scripts/install.mjs

# 3. Start the dashboard
node server/server.js    # http://127.0.0.1:8123

What npm run setup does (install.mjs)

  • A. config.json — copies config.example.json → config.json at the repo root if it does not already exist.
  • B. Statusline tap — backs up ~/.claude/settings.json, saves the previous statusLine.command to ~/.glmps/state/statusline-delegate.json, and points the statusline at taps/statusline-chain.js. Idempotent.
  • C. Capability-reminder hook — appends a UserPromptSubmit hook running hooks/capability-reminder.js. Idempotent.

node scripts/install.mjs --uninstall reverses B and C (restoring the prior statusline) and leaves config.json in place.

What glmps init adds (glmps-init.mjs)

  1. Runs checkDeps() — exits on a missing required dep, warns on optional.
  2. Discovers installed harnesses (each adapter's detect()) and scans --search dirs (default ~) up to depth 2 for .git repos.
  3. Writes glmps.profile.json in the current directory (errors if it exists unless --force).
  4. Calls the same A/B/C wiring as npm run setup.

Running & restart rules

  • Command: node server/server.js (or npm start). Binds 127.0.0.1 only — localhost, never 0.0.0.0.
  • Run it from the directory containing glmps.profile.json (or set GLMPS_PROFILE).
  • web/ changes are live on browser refresh — no restart. Only server/** and taps/** changes need a restart (kill the PID on 8123, relaunch).
Localhost by design. The server has no authentication and exposes file/secret/launch endpoints, which is why it binds loopback only. Making it network-reachable is a separate, security-gated design (auth + TLS tunnel + endpoint hardening), not a one-line bind change.

The statusline tap

Claude Code pipes a JSON status blob to its statusline command on every tick. taps/statusline-chain.js writes ~/.glmps/status/<sid>.json (the live status the server polls), appends an NDJSON usage record (model, cost, durations, tokens, cache, context %, cwd) to ~/.glmps/usage/<date>.ndjson, then spawns the previous statusline command — so it composes with any existing tap.

The companion (Antigravity extension)

An unpublished VS Code extension targeting Antigravity. On startup it health-checks 127.0.0.1:<port> and, if down and missionControl.serverPath is set, spawns node server/server.js (re-using Antigravity's Electron binary with ELECTRON_RUN_AS_NODE=1), re-probing every 30 s. It adds a status-bar open button and drains ~/.glmps/requests/resume.jsonl — opening a terminal per request (claude --resume <id> for resumes). Build with npm run build (esbuild) and npm run package (vsce) inside companion/; those dev deps never reach the server.

Key workflows

  • Watch a live session. Open Sessions; the roster auto-loads from /api/state and updates on each SSE events message (2 s debounce). Select a row; the detail pane streams new events in place.
  • Search history. In History, pre-filter by project/tool/date, then full-text search (/api/search). Click a result to open /session/<id>.
  • Resume a session. Click Resume on a History row; /api/resume writes a request and launches Antigravity at the session cwd; the companion opens it.
  • File & queue a card. Create on the board (or POST /api/backlog). It is poison-scanned at intake; clean items land in Backlog, flagged ones are quarantined pending /api/backlog/:id/approve. Drag to Queued; with the runner enabled the next tick launches it.
  • Review a learning candidate. In Learning, read the body and proposed rule; Approve to write it into the project CLAUDE.md, Promote to lift it to global or a memory file, Alternative to edit the rule, Discard to close. Recurrence badges prioritize.

04Capture & the portability chassis

The observe plane is a poll loop (default 1 s) over discovered session sources. It is deliberately mechanical: discover → tail → extract → index → stream.

~/.claude/projects/*.jsonl ┐ ~/.gemini/antigravity/... ├─▶ discover() ──▶ tail (byte offsets) ──▶ extract() ──┐ ~/.codex, ~/.local/... ┘ sessions.js tailer.js adapters/*.js │ ▼ dashboard ◀── SSE /api/events ◀── event ring + IndexStore ◀─┘ index-store.js

The pipeline

  • Discover — discoverClaudeSessions() reads ~/.claude/.claude-manager/active-sessions.json and scans ~/.claude/projects/*/*.jsonl; discoverAgSessions() walks Antigravity brains; every other tool returns descriptors from its adapter's discover(P).
  • Tail — three source kinds per tick: jsonl-tail (byte-offset incremental read, offsets persisted in state/offsets.json), json-snapshot (mtime-triggered full re-read), sqlite-steps (drains new rows past a cursor). First sight backfills the last 2 MB; nested subagents/*.jsonl tail under the parent session.
  • Extract — extractClaudeEvents(line, sessionId) parses each line and maps every tool_use through classify() into the shared event shape. Foreign adapters normalize native tool names (e.g. gemini's read_file → Read).
  • Index — IndexStore keeps a Map<sessionId, Record> (eventCount, lastTs, skillsUsed, title, cwd, model) and atomically flushes state/index.json when dirty.
  • Stream — pushEvents() strips kind:'meta' events, appends to a 2000-event ring, applies to the index, and writes to every SSE client.

The adapter contract

Every module in server/lib/adapters/ exports a uniform interface, so adding a tool is adding a file:

ExportPurpose
id, displayNameStable identifier + label.
controllableBoolean. true only for claude-code — the act gate keys on this.
detect(P){installed, dataDirs} — presence check.
discover(P)Returns SourceDescriptor[]: {id, tool, kind, file, cwd, mtimeMs, ...}.
extractLine / extractSnapshot / extractStepsPer source kind; return events (+ title/cwd).
tierDirs(P) optionalContributes brain/ephemeral paths to the tier model.

Deep adapters (10): claude-code, antigravity, gemini-cli, copilot-chat, codex-cli, openclaw, hermes, agy-cli, opencode, cline. Detect-only (6): cursor, windsurf, aider, continue, zed, copilot-cli.

The shared event shape

The connective tissue of the whole system — every adapter emits it, every consumer (index, SSE, detectors, UI) speaks only it.

{
  kind:      'tool'|'skill'|'agent'|'mcp'|'context-file'|'memory'
           | 'file-edit'|'command'|'git'|'thinking'|'tokens'|'meta',
  lane:      'feed' | 'context',     // activity stream vs durable context
  label:     string,
  path:      string | null,
  tool:      string,                 // 'Bash', 'Edit', 'mcp__discord__reply', ...
  ts:        number | string | null,
  sessionId: string,
  op?:       'read' | 'write',        // context-file / memory
  change?:   { hunks: [...] },        // Edit/Write diff
  model?:    string | null           // agent dispatch / thinking
}

kind:'meta' events (cwd discovery, model capture) are applied to the index but never streamed. lane:'feed' is the temporal activity stream; lane:'context' is durable cognitive context (skills, agent dispatches, MCP, memory writes, CLAUDE.md reads).

The portability chassis: getPaths + the three-tier model

getPaths(env, profile) resolves every harness/state/asset path with the env > profile > default precedence. After it returns, startServer() sets P.zoneConfig.tierRoots = deriveTierRoots(P, adapters) so live classification sees the tiers.

TierDefault rootsPolicy
Artifact (1)repoRoots — the client's sourcegate — read freely, mutate only through a gate
Brain (2)assetsDir, agentsDir, claude-code skills/ agents/ hooks/versioned — edit, but commit
Ephemeral (3)stateDir, claude-code projects/ cache/readonly — telemetry

The tier axis is orthogonal to the dev/prod zone axis: classifyPath() returns both a {zone, env, protected} and a tier. Tier classification is additive — it never weakens the existing prod-protection. sessionScope() aggregates a session's edited paths into per-zone and per-tier counts, which power the scope strip.

06The self-improvement loop

This is what makes GLMPS more than a viewer. The same parser that drives the live dashboard also drives an offline learning loop: observed mistakes become reviewable guard rules.

session ends ─▶ Stop hook (capability-feed.js) ─▶ extractClaudeEvents() ─▶ feedFromTranscript() │ ┌───────────────────────────────────────────┼──────────────────────────────┐ ▼ ▼ ▼ detectGaps() (event) scanTranscriptForGaps() (raw) deferral ─▶ backlog ticket └──────────────── merge by code ──────────────┘ (poison-scanned) ▼ learning-store.js (upsert, dedup by sha1(code|project)) ▼ rule-synth.js (regex oracle) + cross-model-eval.js (critic) ─▶ human review (/api/learning) ▼ approve / promote ─▶ learning-apply.js ─▶ insertGuard() into CLAUDE.md / CLAUDE.global.md / memory

Detectors

Two complementary detectors run on every ended session and merge deduped by code (first wins):

  • detectGaps() (gap-detect.js, event-shape): ui-without-frontend-design, heavy-edits-no-subagents (≥15 edits, no agent), ui-design-too-late, reread-loop (>8 reads of one path), done-without-verification.
  • scanTranscriptForGaps() (transcript-gaps.js, raw JSONL): bash-grep-over-grep-tool, opus-on-mechanical (subagents with no model set), serial-agents-no-parallel, sleep-poll, backtest-result-without-skeptic (opt-in via backtestProjects), reread-loop.

Store, oracle, and apply

  • Store — learning-store.js upserts each gap keyed sha1(code|project), bumping recurrence per distinct session. Items carry a proposedGuard (file + section + rule template). States: pending → applied | discarded | dispatched | failed; discarded/applied never resurface.
  • Oracle — rule-synth.js validateRule(regex, missed, heldOut) requires the rule to match at least one missed prompt and zero held-out prompts before it can be committed. cross-model-eval.js adds an independent Haiku critic for a 0–1 disagreement score.
  • Apply — approve calls applyGuard() → idempotent insertGuard() into the assets CLAUDE.md, git-commit, mark applied with the SHA. promote targets CLAUDE.global.md or a memory file (dispatched as a claude -p instruction the companion executes).
  • Weekly sweep — scripts/capability-synth.mjs walks all transcripts and upserts gaps; capability-synth-tick.js (SessionStart) launches it detached if the watermark is >7 days old.

The backlog board & queue runner

The board is a tracker; the runner is the opt-in worker. pickNextJob() selects the lowest-order queued & non-quarantined item; shouldClaim() gates on enabled/paused/concurrency. A card flows queued → in_progress (sessionId attached) → done when the launched session self-reports agent:done. worktreePlan() isolates parallel jobs in glmps-runner/<proj>/<jobId> worktrees when concurrency > 1. All git recipes are pure {file, args} objects executed without a shell — safe for spaced paths.

The outcome / critic loop

A higher tier records end-to-end session outcomes for champion/challenger evaluation. session-outcome.js builds an OutcomeRow from classifyTask() (task class), computeMetrics() (turns, wall clock, tool errors, context ratio, first-try), verifierFromEvents() (test/lint/build pass/fail from Bash result pairs), and acceptance coverage. Rows append to outcomes/rows.ndjson. promotion.js uses Pareto dominance (paretoDominates) so a regression on any metric blocks promotion. A replay/eval set (replay-set.js) provides the fixed task suite for challenger runs.

07Safety & governance

The threat this project is built to close: injected content becoming persisted state or an executed job. The defenses are layered so an attack is stopped at the earliest possible point.

Defense-in-depth layers

#LayerWhat it stops
0capability-reminder.js (UserPromptSubmit)Prevents the miss up front: injects ≤3 skill/tool reminders when the prompt matches a rule.
1Poison-scan at intake (poison-scan.js + backlog-store.js)Quarantines a ticket before it enters the queue. Critical flags quarantine regardless of source; any flag quarantines external-trust content (closes the Stop-feeder → ticket → auto-run amplification loop).
2Queue execution (queue-runner.js)Refuses to launch a quarantined card; "Run now" returns 409. Release only via /approve.
3File-API secret denylist (file-api.js)Denies reads/writes of credential/secret files even inside an allowed root — blocks the confused-deputy exfil path.
4Graph-rebuild allowlist (server.js)Validates body.root against known graph roots before a shell:true spawn — blocks shell injection.
5Memory-integrity scan (memory-scan.js)Content scan + SHA-256 manifest diff surfaces planted/drifted memory entries before they steer future sessions.
6Live-install guard (live-install-guard.js, PreToolUse)Blocks writes into the live trading install (GLMPS_LIVE_INSTALL_PATHS); bypass needs GLMPS_ALLOW_LIVE_INSTALL=1 or a live-install.allow file.
7Scope-guard + done-gate (Stop)Blocks "done" if prod-zone files were touched, or if acceptance.md commands fail (capped at 3 attempts so an agent is never trapped).
8Act gate (act-gate.js)Only the controllable harness (Claude Code) can actuate; foreign harnesses are structurally observe-only.

The poison-scan detector table

CodeSeverityMatches
unicode-tagcriticalU+E0000–E007F hidden tag chars.
instruction-overridecritical"ignore previous instructions", "your new instructions:".
authority-spoofcritical"you are now", "act as", "developer mode", "comes from Anthropic/the system".
exfiltrationcriticalsend/post/upload to a URL or email; "include api_key / .ssh in...".
credentialcriticalsk-, AKIA, ghp_, xox, AIza, PEM headers, bare JWTs.
zero-width / memory-injection / external-url / base64-instructionwarnZWJ/BOM; "remember that / from now on"; bare URLs; instruction-like base64.

classifyTrust() tags operator-typed content as operator and everything else as external; quarantineDecision() applies the asymmetric policy above.

Dev/prod separation

classifyPath() returns env:'prod' for paths under a protectedRoots entry or with a prod/-prod segment. The six prod surfaces (per docs/dev-prod-separation.md): live trading installs, the published mirror, FROZEN research files, the running service + real state, secrets, and main. The only legitimate dev→prod crossing is an explicit gate (export script, review, prod.allow, or a backtest-skeptic verdict). The scope-guard activates only in repos that opt in via acceptance.md. See also docs/agent-poisoning-safeguards.md.

Hooks fail open. Every hook is a thin shell over a pure, tested server/lib core and exits 0 on any internal error — a guard bug must never trap the user.

08graphify integration

GLMPS uses graphify to maintain an AST-level knowledge graph of each project (zero API cost). Artifacts live in graphify-out/: graph.json (nodes + edges), GRAPH_REPORT.md (god nodes, communities), and a wiki/.

  • Server side — code-graph.js loadGraph() reads the graph and overlays zone config; graph-status.js compares the graph's builtFromCommit to HEAD for freshness.
  • Endpoints — GET /api/graph (load for a project), GET /api/graph/status (stale/fresh badges), POST /api/graph/rebuild (runs graphify update; root allowlisted; spawned detached so worker subprocesses don't pop console windows).
  • Proactive nudges — capability-reminder.js tells the agent to query graphify before re-reading files when a prompt is architecture/orientation-shaped; capability-feed.js auto-runs graphify update after any session that touched code, keeping the graph current with no manual step.

09How it all coheres

The system is two flows sharing one spine. The spine is the shared event shape: one parser (extractClaudeEvents) feeds the live server, the Stop-hook feeder, and the weekly sweep, so observation and learning never diverge.

┌───────────────────────────── OBSERVATION FLOW ─────────────────────────────┐ │ agent sessions (any harness) │ │ │ write transcripts │ │ ▼ │ │ capture adapters ─▶ classified events ─▶ IndexStore + ring ─▶ SSE ─▶ dashboard └──────────────────────────────────┬───────────────────────────────────────────┘ │ same events, same parser ┌──────────────────────────────────▼────────────── LEARNING FLOW ────────────┐ │ Stop hook ─▶ detectors ─▶ learning store ─▶ oracle/critic ─▶ human review │ │ │ │ │ ▼ │ │ guard rules back into CLAUDE.md / memory │ │ (deferrals ─▶ backlog ─▶ queue runner ─▶ launch a new Claude Code session) │ └───────────────────────────────────────────────────────────────────────────────┘ ▲ ▲ ▲ │ │ │ poison-scan at file-api denylist / live-install guard / backlog intake graph-rebuild allowlist scope + done gate (safeguards wrap intake, exec, and the file API)

In words: agents work; GLMPS watches them through adapters and shows the live picture. When a session ends, the same event stream is replayed offline to find capability misses, which become reviewable guard rules that flow back into the agent's own instructions — so the next session is a little less likely to repeat the mistake. Wrapping all of it, a defense-in-depth layer ensures that nothing injected into a transcript, a memory file, or a backlog ticket can quietly become persisted state or an executed job. The portability chassis (engagement profile + three-tier model + the observe/act split) is what lets this whole apparatus be pointed at any client's environment instead of one hard-wired machine.
